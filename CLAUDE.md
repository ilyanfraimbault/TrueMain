# TrueMain — working conventions

TrueMain is a League of Legends analytics site: champion/player stats computed from Riot API data.

## Repo map

- `backend/` — .NET solution (`TrueMain.sln`): `Api` (REST), `Ingestor` (Riot data pipeline), `Data` (EF Core → PostgreSQL, plus Mongo for metrics), `Core`, `tests/`.
- `web/` — public Nuxt + Nuxt UI frontend.
- `admin/` — standalone Nuxt admin portal (separate app, not a `/admin` route).
- `compose*.yaml` — Docker stacks. **Prod deploys from the versioned `compose.prod.yaml`, but deploys are manual** (`git pull` + `docker compose -f compose.prod.yaml up -d` on the VPS) — merging infra changes does not reach prod by itself; deployment must be called out explicitly.
- `docs/` — project docs.

## Language

- Talk to the user in French.
- Everything committed or published — code, comments, commit messages, issues, PR titles/bodies — is in English.

## Git & PR workflow

- Default branch: `develop`. All PRs target `develop`; the only PR allowed to target `master` is the release PR from `develop`.
- Branch names: `<type>/<issue>-<short-kebab>` (e.g. `feat/526-rank-scoped-builds`). Rename autogenerated worktree branches before the first push. Never a `claude/` prefix.
- Conventional commits, no `Co-Authored-By: Claude` trailers, no "Generated with Claude Code" footers on PRs.
- PR body contains `Closes #N` to auto-link the issue.
- Feature PRs are **squash**-merged into develop; release PRs (develop→master) use a **merge commit** (squash there creates false conflicts on the next release).

## Merge policy — finish the job

A PR is **done** when, on its current HEAD: CI is green, the Claude review verdict is clean (check the review's `commit_id` against the head SHA — verdicts trail ~1 commit behind pushes), and every real finding is fixed or rebutted in a comment.

- When done: **merge without asking** and delete the branch (`gh pr merge --squash --delete-branch`). The user has delegated the merge decision.
- Never delete `develop` or `master`; develop survives release merges.
- Don't push cosmetic-only commits to a PR that is already done. After ~3 fix iterations that don't converge, stop and report the blockers instead of looping.

## Project board

Every issue goes on GitHub Project #2 ("TrueMain"). Status: Todo / In Progress / Done. Priority = sprint bucket: P0 current, P1 next, P2 after, P3 someday (no milestones). Update fields with `.claude/scripts/project-set.sh <issue> <Status|Priority> <value>`.

## Build & verification (CI traps)

- Backend: CI builds **Release** with code-style analyzers as errors — run `dotnet build backend --configuration Release` before pushing backend changes, not just Debug.
- Schema changes: regenerate the EF compiled model (`dotnet ef dbcontext optimize`, output in `Data/CompiledModels`). Two schema PRs merged concurrently always conflict there — the second one must re-merge develop and regenerate (a stale model silently drops columns).
- Frontends: `nuxt typecheck` can pass on stale `.nuxt` types while CI's `nuxt build` fails — verify type-level changes with a fresh build.
- `web/package-lock.json`: regenerate with `npx npm@11.13.0` (CI's npm version; older npm omits sharp optional deps).
- Startup migrations must stay fast — a heavy `CREATE INDEX CONCURRENTLY` in a startup migration blows the command timeout and crash-loops the API.
- API reads live in `Data` as purpose-built query objects returning read-models; `Api` stays persistence-ignorant. No generic `IRepository<T>` for reads.

## Skills

Prefer the project skills for lifecycle steps — they encode all of the above: `start-issue`, `ship` (verify → PR → babysit → autonomous merge), `new-issue`, `release`, `cleanup-branches`, `setup-worktree`.
