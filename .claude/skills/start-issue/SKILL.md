# Start an issue

1. **Read it fully**: `gh issue view <n> --comments`. Follow references to linked issues/PRs — design decisions often live in a parent epic or a prior PR discussion.

2. **Board**: `.claude/scripts/project-set.sh <n> Status "In Progress"` (adds the issue to Project #2 automatically if it isn't on the board).

3. **Branch** — `<type>/<n>-<short-kebab>` with type ∈ feat, fix, perf, refactor, chore, docs:
   - In a fresh worktree, rename the autogenerated branch: `git branch -m <type>/<n>-<short-kebab>`.
   - Make sure the branch starts from up-to-date develop: `git fetch origin develop`, then check `git log HEAD..origin/develop --oneline`. If the worktree was cut from master or a stale develop and has no local work yet, `git reset --hard origin/develop`.

4. **Plan before coding**: identify the layers touched (web/admin → Api → Data). API reads belong in `Data` as purpose-built query objects returning read-models — no generic repositories. Schema changes imply an EF migration + compiled-model regeneration (the `ship` skill re-checks this).

5. When the work is implemented and verified, invoke the `ship` skill to land it — it handles PR, review babysitting, autonomous merge and branch cleanup.
